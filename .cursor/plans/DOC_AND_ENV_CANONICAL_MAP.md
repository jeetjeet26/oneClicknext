# Documentation And Env Canonical Map

Last Updated: March 17, 2026

## Purpose

Reduce duplicate guidance and conflicting setup instructions by defining one canonical source for planning docs and environment configuration.

## Canonical Planning Sources

- Execution order and current status: `.cursor/plans/PROJECT_IMPLEMENTATION_GUIDE.md`
- Phase roadmap and acceptance gates: `.cursor/plans/AUTONOMY_FOUNDATION_ROADMAP.md`
- Baseline gap audit snapshot (historical): `.cursor/plans/AUTONOMY_FOUNDATION_CODEBASE_GAP_REPORT_2026-03-16.md`
- Vision intents by domain: `.cursor/plans/visions/*.md`

## Legacy / Archived Planning Sources

- Archived specs and runbooks: `outdateddocs/**`
- Treat these as reference history, not live execution truth.

## Canonical Environment Sources

- Shared template for local and hosted setup: `p11-platform/.env.example`
- Shared runtime secrets file (local machine only): `p11-platform/.env`
- Generated local Supabase overlay: `p11-platform/.env.local` (generated by `npm run supabase:reset`)

## Service Env Policy

- Web app and data-engine both read from shared root env files in `p11-platform/`.
- Do not add a separate canonical template under service folders unless variables are truly service-exclusive and cannot be expressed in the shared root template.
- Keep route/service config names consistent (`DATA_ENGINE_URL`, `NEXT_PUBLIC_SITE_URL`, etc.) and avoid introducing alias env names without a migration plan.

## Maintenance Rule

When README instructions conflict, update this map and the canonical source first, then prune or rewrite duplicated instructions in secondary docs.
